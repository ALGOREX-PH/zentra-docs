The verifier runs a full Groth16 check **inside the Soroban contract** using
Protocol 25/26 BN254 host functions. No trusted relayer, no off-chain verifier.

[**Try it in the playground**](/playground) — generate a real Groth16 proof in your
browser, inspect the 14 public signals, and anchor it on-chain through your wallet:

![Generating and anchoring a real Groth16 proof in the Zentra playground](/img/playground-anchor.png)

## The proof

```rust
struct Proof { a: G1Affine, b: G2Affine, c: G1Affine }
```

On the wire it's a **256-byte** blob — `a (64) ‖ b (128) ‖ c (64)` — with every
coordinate a 32-byte big-endian field element.

## Aggregating the public inputs

The verifying key carries **15 IC points** (one constant, plus one per public
input). The contract folds the 14 public signals into a single point with a
multi-scalar multiplication:

```text
vk_x = IC[0] + Σ pub[i] · IC[i + 1]
```

## The pairing check

Verification is one multi-pairing equation over BN254:

```text
e(-A, B) · e(α, β) · e(vk_x, γ) · e(C, δ) == 1
```

If it holds, the proof is valid and the contract proceeds to settle. If not, it
returns `InvalidProof` and nothing moves.

<Callout title="The cost">
The whole verification — the MSM plus the multi-pairing — measures at about
<CpuBudget /> of the per-transaction CPU budget on testnet. On-chain ZK
verification isn't just possible on Soroban; it's comfortably affordable.
</Callout>
